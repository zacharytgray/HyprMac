## Subsystem summary
New-window assignment as-coded: the 1Hz PollingScheduler (PollingScheduler.swift:25) or a notification-driven schedule (appDidLaunch +0.5s, WindowManager.swift:1842) triggers pollWindowChanges (WindowManager.swift:1712), which snapshots AX via AccessibilityManager.getAllWindows and diffs in WindowDiscoveryService.computeChanges. New windows are assigned in ActionDispatcher.applyChanges -> assignNewWindow (ActionDispatcher.swift:196-207) by the window's PHYSICAL screen — center of a live AX frame read (HyprWindow.swift:128-148) resolved through DisplayManager.screen(at:) which has a nearest-screen fallback that always returns some screen — and the window joins whatever workspace is currently VISIBLE on that screen; the cursor screen is only a fallback when the AX frame is unreadable. This diverges from the crib in three ways: (1) AX->CG window-ID matching is SPI-first (_AXUIElementGetWindow) with greedy best-fit only as a fallback (AccessibilityManager.swift:171-210), (2) the workspace-transition poll suppression is 1.5s (WorkspaceOrchestrator.swift:79,159) not the 0.6s the WindowManager comment claims (WindowManager.swift:176-177), and (3) drift detection now unconditionally skips any window whose recorded workspace is hidden (WindowDiscoveryService.swift:327) — the hasSubstantialOnScreenFrame helper built to qualify that case is dead code (WindowDiscoveryService.swift:341). Gone windows whose pid is still alive move to hiddenWindowIDs and are proactively removed from their BSP tree with compaction (ActionDispatcher.swift:122-124, TilingEngine.swift:83-97); on return they are smart-inserted as brand-new leaves (TilingEngine.swift:343-350), so any transient disappearance permanently loses the window's slot. Crucially, NO polling suppression exists on wake/lock/unlock — systemInterruption only resets hotkey/mouse state (WindowManager.swift:1828-1837) — so post-wake polls run against half-awake AX/CG snapshots at full cadence, which is the only chain found that produces wake-scramble in single-monitor laptop mode.

## Subsystem summary
The discovery-drift subsystem works as follows as-coded: PollingScheduler runs a 1Hz timer plus coalesced notification polls (drop-when-pending, suppression re-checked at fire time — no in-flight race, everything is synchronous main-thread). Each poll takes a full AX snapshot (per-app kAXWindowsAttribute queries in app-enumeration order; any app whose AX query fails silently contributes zero windows), then WindowDiscoveryService.computeChanges diffs it against WindowStateCache: present-and-hidden ids become 'returned' (workspace assignment and floating state retained, tree position NOT — the node was destroyed when it went hidden), unknown ids become 'new' (assigned by physical center-point screen with cursor fallback, then immediately tiled), missing ids become gone→hiddenWindowIDs when the pid is alive or fully forgotten when dead, and a stale-state sweep forgets orphans (caller cleanup via applyForgottenIDCleanup does remove workspace assignments, so no leak). Drift detection reassigns a tiled window when its center-point screen's visible workspace differs from its recorded workspace, guarded by three skips: floating, home-screen match, and an unconditional 'recorded workspace hidden' skip; the hasSubstantialOnScreenFrame helper that the doc comments describe as the park-position defense is dead code — parked-frame protection now rests entirely on the hidden-ws skip plus the time-boxed 'workspace-transition' suppression (1.5s on workspace switch/move, 2.0s on display change — the WindowManager comment saying 0.6s is stale, and SuppressionRegistry's own doc omits the workspace-transition key entirely). The critical structural gaps versus the architecture crib: (1) nothing suppresses discovery across sleep/lock/wake, so partial wake snapshots churn gone→returned and rebuild trees in snapshot order; (2) 'non-destructive' reconcile accepts transient wake topologies as real because the fingerprint only drops identical layouts and the 0.5s settle is far shorter than display reattach; (3) the returned path has no slot memory and no correction for recycled-CGWindowID reopens stranded on hidden workspaces. Suppression durations are fixed dates racing an AX staleness the code itself documents as unbounded above 1 second.

## Subsystem summary
The concurrency picture as-coded: everything runs on the main thread — the CGEventTap is attached to the main run loop (HotkeyManager.swift:92), NSEvent global monitors and NSWorkspace/DistributedNotificationCenter observers fire on main, and PollingScheduler's Timer plus all asyncAfter blocks resolve on main — so there are no true data races; every bug in this subsystem is an interleaving of deferred async stages. Entry points that end in tileAllVisibleSpaces/reconcile are: the 1Hz timer, coalesced notification polls (activate/launch/terminate/hide), screenParametersChanged's 0.5s-delayed reconcile, hotkey actions via ActionDispatcher, drag-end via DragSwapHandler's 0.1s settle, config sinks, forgetApp, and start()'s 1.0s-delayed snapshotAndTile. The only reentrancy guards between them are date-gated suppression keys ('workspace-transition' 1.5-2.0s, 'cross-swap-in-flight' 0.8s) checked by PollingScheduler at schedule AND fire time — that double-check is correct, but several windows are simply uncovered: wake (systemInterruption resets only hotkey state, never suppresses discovery), the [mouseUp, +0.1s] drag-settle gap, and the gap between suppression expiry and late display attach. The architecture crib's claim that 'discovery polling is suppressed through the settle window' is therefore only true for the happy path where all display notifications arrive within 2s and windows settle instantly. A second divergence from the crib: the fingerprint guard on didChangeScreenParameters treats 'screen list unchanged' as 'nothing happened', but sleep can move windows without changing the final screen list, and skipping the reconcile leaves drift detection to convert those OS moves into permanent workspace reassignments. The dominant scramble engine, valid even in single-monitor mode, is the discovery gone→hidden→returned round-trip: one transient AX dropout removes the window from its BSP tree with an immediate full compact (dropping split overrides and slot positions), and the return re-enters via smartInsert at a fresh slot, in AX z-order for batches — so layout 'memory' does not exist anywhere in the system; tree shape is a pure function of discovery event ordering. PollingScheduler's coalescing also drops longer requested settle delays when a shorter poll is pending, undermining the app-launch settle. Finally, stop()/restart() leaves both a queued coalesced poll and the previous start()'s deferred snapshotAndTile alive with no generation token, and pollWindowChanges never checks isRunning.

## Subsystem summary
The persistence-memory subsystem is, as-coded, almost entirely absent: ConfigStore/ConfigMigration persist only user preferences (keybinds, gaps, colors, per-monitor settings), while every piece of layout state — window→workspace assignments and the per-screen visible-workspace map (WorkspaceManager), BSP tree topology/ratios/overrides (TilingEngine/BSPTree), the floating set, originalFrames, and savedFloatingFrames — lives exclusively in process memory. The survival matrix: workspace assignments survive retiles, workspace hide/show, app hide (via hiddenWindowIDs), and display reconfig (reconcileAfterDisplayChange preserves them; classifyAndAssign only assigns unassigned windows), but are rewritten by drift detection, by assignNewWindow on CGWindowID recycling, and destroyed wholesale by distributeWindowsAcrossWorkspaces. Tree topology survives hide/show of its workspace but is rebuilt from scratch (compact) on ANY window close or minimize; split-ratio memory (userSetRatio) survives pure retiles — the docs' claim that ratios always reset to 0.5 is stale — but is cleared tree-wide on any insert/remove; togglesplit overrides die in compact. The manual float set survives everything except distributeWindowsAcrossWorkspaces, which un-floats all non-excluded windows. Nothing survives a HyprMac restart or an enabled off/on toggle, because start() unconditionally runs snapshotAndTile → distributeWindowsAcrossWorkspaces; contrary to the stated 'launch and Retile All only' contract, that destructive path is also reachable from the maxSplitsPerMonitor config sink and from an enabled flip arriving via iCloud file sync. Display migration snapshots nothing: trees are re-keyed by recomputed home (dropping the smaller tree on key collision), the visible-workspace map is pruned whenever a screen's origin-derived screenID changes, and hidden windows are re-parked — but wake itself (didWakeNotification) triggers only a hotkey-state reset, with no discovery suppression and no repair pass, so the first post-wake polls act on OS-shuffled window positions. The only snapshot machinery in the codebase, BSPTree.Snapshot, is in-memory, non-topological, and used solely for speculative-swap rewind; there is no dead persistence code to resurrect and the failed-fix patch contains no prior persistence attempt. Net: the system treats window positions as authoritative and layout state as disposable, which is exactly backwards for the author's layout-memory requirement.

The wake/display subsystem works as follows as-coded: sleep/wake/lock notifications (didWake, screensDidWake, sessionDidResign/BecomeActive, screenIsLocked/Unlocked) are all routed to WindowManager.systemInterruption (WindowManager.swift:1828), which ONLY resets hotkey/dock/menu-tracking state — it sets no discovery suppression and triggers no reconcile, so the 1Hz poll (PollingScheduler) keeps running through the entire wake sequence. ## Subsystem summary
The retile pipeline as-coded diverges from the architecture crib in five load-bearing ways. (1) The per-retile diff (TilingEngine.updateTreeMembership) has zero hysteresis: any window missing from the single AX snapshot passed in is removed from its tree immediately, and ANY removal triggers BSPTree.compact — a full from-scratch rebuild of the whole tree, not the incremental prune the crib implies. (2) There are two different insertion algorithms: live inserts go through LayoutEngine.smartInsertFitting (consults MinSizeMemory via pairFits), while compact's rebuild uses BSPTree.smartInsert (geometry only, no min-size checks) — so a rebuild can produce a different topology than live insertion would, and compact unconditionally destroys every togglesplit override and user ratio. (3) Batch insertion order is the AX enumeration order of getAllWindows (runningApplications x kAXWindowsAttribute), which varies with per-app window MRU/z-order — any cycle that inserts 2+ windows produces an order-dependent, run-to-run-varying tree shape. (4) Trees are keyed by (workspace, screen-origin-coordinates) rather than displayID; handleDisplayChange migrates whole trees on reconcile and on key collision drops the smaller tree wholesale ("prefer larger"), whose windows are then batch-reinserted in AX order. The DUP diagnostic in intendedTileRects and the doc comment in handleDisplayChange confirm the two-trees-one-window state occurs in production. (5) The two-pass readback (FrameReadbackPoller) sleeps the MAIN THREAD up to 0.36s per pass per screen, and its "settled conflict" classifier cannot distinguish an app that refuses to shrink from an app that simply hasn't responded yet — a stable stale frame ratchets MinSizeMemory up (documented in-code as the Tahoe race; auto-float-on-overflow was already disabled because of it), while an AX read failure falls back to the target size and is classified as "accepted", which can falsely lower a correct minimum. Nothing anywhere records slot assignments persistently: once a node is evicted (transient AX blackout after wake, Cmd+H, dropped tree), the arrangement is unrecoverable by design — reinsertion is pure heuristic placement. Also of note: distributeWindowsAcrossWorkspaces genuinely only runs at startup/Retile All (crib accurate on that), and reconcileAfterDisplayChange runs once per fingerprint-passing screenParametersChanged fire, meaning a staged monitor reattach at wake runs the full migrate/repark/retile dance 2-3 times against transient configs.

## Subsystem summary
WorkspaceManager is a pure bookkeeping layer: monitorWorkspace ([screenID: workspace]) records which workspace each screen shows, windowWorkspaces/workspaceWindowSets record assignments, and screenID is a hash of the NSScreen frame origin (x*10000+y). Static anchoring works as documented: homeScreenForWorkspace is enabledScreens[(N-1)%count] over left-to-right enabled screens, and initializeMonitors enforces the invariant that a screen only shows a workspace anchored to it, falling back to the lowest anchored workspace and deleting entries whose sid no longer matches a live screen — which means any origin/topology change silently discards which workspace each monitor was showing. The park mechanism is a single global corner (rightmost of ALL screens, including disabled ones) at (visibleFrame.maxX-1, maxY-1); hideInCorner ignores its `screen` parameter entirely, so WorkspaceOrchestrator's comments about "parking on the workspace's static home monitor" are doc drift — everything parks at the one global corner. Parking is fire-and-forget: setPositionOnly discards the AXError, there is no readback, and nothing in steady state ever re-parks a hidden window (re-park runs only inside reconcileAfterDisplayChange, which the display fingerprint skips when the layout string is unchanged); drift detection deliberately never inspects hidden-workspace windows, so a parked window the OS moves back on-screen just sits there. WorkspaceOrchestrator.switchWorkspace matches the crib (suppress 1.5s, hide old via park, restore floater frames, retile, focus best), but moveToWorkspace derives the window's "current workspace" from its physical screen (center-point lookup) rather than from windowWorkspaces, so its tree-removal and same-workspace guard can target the wrong tree. New-window assignment (ActionDispatcher.assignNewWindow) is physical-position based with a nearest-screen fallback, which the park corner biases toward the rightmost monitor. Wake handling diverges from the crib in one important way: systemInterruption (didWake/screensDidWake) only resets hotkey state — discovery suppression starts only when didChangeScreenParameters is processed, leaving a window at wake where the 1Hz poll can drift-reassign OS-moved windows against stale monitorWorkspace state.

### [UNVERIFIED] No discovery suppression at wake: poll can drift-reassign OS-moved windows against stale monitorWorkspace before the screen-change notification is processed
- **Where**: HyprMac/Core/WindowManager.swift:1828  |  symptom: wake-scramble  |  auditor confidence: medium  |  fix size: small
- **Mechanism**: 1. During sleep an external display drops and macOS relocates its windows onto the surviving screen(s) before HyprMac runs again. 2. At wake the 1Hz discovery poll can fire before didChangeScreenParameters is processed: systemInterruption (WindowManager.swift:1828-1837) sets no suppression, and the only poll gates are cross-swap-in-flight/workspace-transition (WindowManager.swift:178-182, PollingScheduler.swift:49); suppression starts only inside screenParametersChanged (WindowManager.swift:1878). 3. detectScreenDrift computes visibleWorkspaces from the stale monitorWorkspace (WindowDiscoveryService.swift:301), so the dropped screen's workspace still counts as visible and its windows are not exempted by the hidden-ws guard (WindowDiscoveryService.swift:327). 4. Those windows now physically sit on the surviving screen, so physicalWs != recordedWs (WindowDiscoveryService.swift:307-328). 5. ActionDispatcher.applyChanges applies moveWindow for every drifted window (ActionDispatcher.swift:127-129), dumping the entire external monitor's workspace into the laptop's visible workspace, then retiles them in list order — full assignment scramble that reconcileAfterDisplayChange cannot undo.
- **Suggested fix**: In systemInterruption, suppress 'workspace-transition' (e.g. 3-5s) on didWakeNotification / screensDidWakeNotification / screenIsUnlocked, so the first polls after wake cannot run drift until a reconcile has refreshed monitorWorkspace. The one unverified hop is the poll-beats-notification timing; a .notice log at poll entry during the wake window would confirm it per the project's debugging workflow.

### [UNVERIFIED] Park has no self-repair: a hidden-workspace window the OS (or its app) moves back on-screen stays fully visible forever in steady state

- **Mechanism**: 1. hideInCorner parks once at hide time with no readback (WorkspaceManager.swift:249-253, HyprWindow.swift:191-196). 2. ### [UNVERIFIED] monitorWorkspace keyed by frame-origin hash: any origin/topology change discards which workspace each monitor was showing
